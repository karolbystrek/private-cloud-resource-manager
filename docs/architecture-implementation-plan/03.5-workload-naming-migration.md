# 03.5. Future Workload Naming Migration

## Goal

Defer the product naming migration from `Job`/`Run` to `Workload`/`WorkloadRun` until the state model is stable.

The current implementation work should continue using the existing `job` and `run` names. This document records the
target vocabulary so later refactors do not mix product entities with Nomad scheduler identifiers.

## Target Naming

Use product names for internal platform entities:

- `workload`: user-submitted workload intent, replacing product `job`.
- `workload_run`: one execution attempt of a workload, replacing product `run`.

Use explicit Nomad-prefixed names for external scheduler identifiers:

- `nomad_job_id`: external Nomad job ID.
- `nomad_allocation_id`: external Nomad allocation ID.
- `nomad_eval_id`: external Nomad evaluation ID.

The `nomad_job_id` field must never be confused with an internal product ID. It is owned by Nomad naming and lookup
contracts, not by the product aggregate model.

## Naming Rules

- Never use bare `jobId` for Nomad identifiers.
- Internal product IDs should be named `workloadId` and `workloadRunId`.
- External scheduler IDs should be named `nomadJobId`, `nomadAllocationId`, and `nomadEvalId`.
- Database columns for external scheduler IDs should keep the `nomad_` prefix.
- Nomad job names should remain deterministic:
  - `pcrm-run-<workload_run_id>`
  - or `pcrm-workload-run-<workload_run_id>`

## Migration Phases

### 1. Documentation and API Language

Update architecture docs and user-facing frontend labels from "job" to "workload" where the product concept is meant.
Keep Nomad-specific language unchanged when referring to Nomad jobs, allocations, and evaluations.

### 2. Database Additive Phase

Introduce compatibility views, renamed tables, or renamed columns only after the run model is stable. Keep this phase
additive first so existing callers and workers can continue operating during the transition.

Potential end-state names:

- `workloads` for user intent.
- `workload_runs` for execution attempts.
- `workload_run_attempts` if low-level dispatch or allocation attempts remain modeled separately.

### 3. Backend Code Rename

Rename Java packages, classes, DTOs, repositories, and services from `jobs`/`Job`/`Run` to
`workloads`/`Workload`/`WorkloadRun`.

Keep the internal/external distinction visible in DTO and entity fields:

- `workloadId`
- `workloadRunId`
- `nomadJobId`
- `nomadAllocationId`
- `nomadEvalId`

### 4. API Compatibility

Keep `/jobs` routes temporarily or add `/workloads` aliases before removing old routes. Compatibility should be explicit
and time-bound, with old names treated as product API compatibility rather than a second domain model.

### 5. Frontend Rename

Update route labels, TypeScript types, component names, and copy from jobs/runs to workloads/workload runs. Keep
operator-visible Nomad labels explicit when displaying scheduler data.

Examples:

- "Workload" for user-submitted intent.
- "Workload run" for one execution attempt.
- "Nomad job ID" only for the external scheduler job identifier.

### 6. Cleanup

Remove compatibility names after no callers depend on them. Cleanup should include database compatibility views or
aliases, old DTO names, old frontend routes or labels, and transitional documentation.

## Deferred By Design

This migration is intentionally out of scope for the current state-model implementation. The immediate priority remains
separating user intent from execution attempts while preserving billing, quota, dispatch, and lease invariants.
